CIRCUITSIM

Michael Costello & Brenna Manning

Design Review 1 Reflection

Feedback and decisions

Overall, we received positive feedback on our project, and people liked our concepts. We gained some helpful input on each of our key questions:

What is the appropriate scope for this project? How in-depth should we go?

Our reviewers liked our focus on a specific user group (iSIM students, in this case) though some said we should keep other intro-EE courses in mind. Reviewers also acknowledged time constraints and agreed that we should begin with some simple MVPs. We will continue to work on the small-scale, developing a simple UI and a simple algorithm for circuit math.

Which specific aspects of circuits should our simulation cover?

Our reviewers agreed that finding basic circuit values is a good place to start, but thought it might be more helpful to do the reverse, calculating resistor/capacitor values for specified voltages and currents. Both basic and complex filters were deemed necessary, but other circuit components like op-amps were suggested as well.

What information would you like the simulator to tell you? (current? voltage?)

Some reviewers suggested that a bode plot (a comparison of voltage oscillation frequency and strength of output) might be more helpful than simply showing predicted voltage values.

Review process reflection

We think the review went well. We got feedback on some of the things we had been unsure about and we think we provided a good amount of context for the audience. We did stick to our planned agenda, but for next time we might consider having a more specific/concrete plan, especially in terms of how to use the space and available resources to our greatest advantage during the review.

FINAL PROJECT PROPOSAL

Educational Application

Big Idea: A program that teaches some aspect of physics, chemistry, or other science, like (PhET). MVP: An interactive program that runs correct science and displays. Stretch Goal: To make our interactive program into a web application to learn about conversion. (This really depends on which concept we decide to do, if we do something like circuit modeling, that could be pretty complicated because there are multiple parts, but this depends on the specific topic - there’s lots of scope!) We would also like to add further features and levels to create a comprehensive tool.

Implementation Plan:

Decide what physics concepts we want to model Code all the physics to demonstrate that the physics works Figure out different “cases” and how we want to demonstrate that. Make into different “levels” or challenges

Project Schedule:

Week 1: Research/Planning - How will we make this program? Choose what we will model and get started on physics demonstration code.
Week 2: Write code to demonstrate more physics cases.
Week 3-4: Get our physics models into an interactive program.
Week 5: add more levels/challenges and additional features.
Week 6: Buffer week for if anything goes wrong.

Collaboration plan:

Working together in the same room but not necessarily pair programming. Working on different aspects of the code, but keeping each other up to date on what is going on, and asking questions when problems come up. Make major decisions as a group.

Learning Goals: web app development, physics-modeling, user-oriented-design

Risks: Pretty low risk - maybe not having enough for three people to do

Additional Course Content: web/mobile development/intro to the languages?
